# Project Structure — Modular Monolith in Go + Gin

## Overview

This project follows a **modular monolith** architecture. Each business domain lives in its own self-contained module inside `internal/`. All shared technical infrastructure lives in `internal/platform/`. Pure utilities with no internal dependencies live in `pkg/`.

The dependency rule is strictly one-directional:

```
cmd/ → internal/*/  →  internal/platform/  →  pkg/
```

No module ever imports another business module. No `platform/` package ever imports a business module.

## File-by-file explanation

---

### `cmd/api/main.go`

**Purpose:** Entry point. Boots the application by wiring all dependencies together.

**What it does:**
1. Loads config from environment variables
2. Opens the PostgreSQL connection pool
3. Connects to Redis
4. Instantiates all module repositories, services, and handlers
5. Registers all routes onto the Gin engine
6. Starts the HTTP server

**Rule:** Contains zero business logic. If you find yourself writing anything other than wiring code here, it belongs in a module instead.

```
main()
  ├── config.Load()
  ├── database.Connect()
  ├── cache.Connect()
  ├── auth.NewHandler(...) → auth.RegisterRoutes()
  ├── user.NewHandler(...) → user.RegisterRoutes()
  └── r.Run()
```

---

### `internal/`

The core of the application. Everything inside is private to this project — Go's module system prevents external packages from importing anything under `internal/`.

---

### `internal/auth/`

**Purpose:** Everything related to registration, login, JWT issuance, and token validation.

| File | Responsibility |
|---|---|
| `model.go` | Structs: `User`, `RegisterRequest`, `LoginRequest`, `AuthResponse`, `Claims` |
| `repository.go` | DB queries: `CreateUser`, `FindByEmail`, `FindByID` using sqlc |
| `service.go` | Business logic: hash password, compare hash, generate JWT, validate JWT |
| `handler.go` | HTTP: parse request body, call service, write response |
| `routes.go` | Registers `POST /api/v1/auth/register` and `POST /api/v1/auth/login` |
| `handler_test.go` | Unit tests for handlers using mock service |
| `service_test.go` | Unit tests for service using mock repository |

**Dependencies:** `internal/platform/database`, `internal/platform/logger`, `pkg/response`, `pkg/crypto`, `pkg/errors`

---

### `internal/user/`

**Purpose:** User profile management for authenticated users.

| File | Responsibility |
|---|---|
| `model.go` | Structs: `Profile`, `UpdateProfileRequest` |
| `repository.go` | DB queries: `GetProfile`, `UpdateProfile`, `DeleteUser` |
| `service.go` | Business rules around profile updates |
| `handler.go` | HTTP: `GET /profile`, `PUT /profile`, `DELETE /account` |
| `routes.go` | Registers all user routes under the JWT middleware group |
| `handler_test.go` | Unit tests for handlers |
| `service_test.go` | Unit tests for service |

**Dependencies:** `internal/platform/database`, `internal/platform/cache`, `pkg/response`, `pkg/errors`

---

### `internal/product/` *(example third module)*

**Purpose:** A placeholder showing how to add any new business domain. Copy the structure from `auth/` or `user/` — the five files are always the same.

Every new feature you add to the app becomes a new folder under `internal/` following the same pattern. The other modules never need to change.

---

### `internal/platform/`

**Purpose:** Shared technical infrastructure. No business logic lives here — only the building blocks that every business module needs.

**Rule:** `platform/` packages may import from `pkg/` only. They never import from business modules (`auth/`, `user/`, etc.).

---

#### `internal/platform/database/postgres.go`

**Purpose:** Opens and manages the `pgxpool.Pool` connection to PostgreSQL.

**What it does:**
- Calls `pgxpool.New()` with the DSN from config
- Pings the database to verify connectivity at startup
- Returns the pool for injection into all repositories

**Used by:** Every module's `repository.go`

---

#### `internal/platform/cache/redis.go`

**Purpose:** Opens and manages the Redis client using `go-redis`.

**What it does:**
- Connects to Redis using host/port/password from config
- Exposes `Get`, `Set`, `Delete`, `Exists` helpers with consistent error handling
- Wraps serialization so callers work with Go structs, not raw bytes

**Used by:** Modules that cache data (e.g. user profile cache, rate limit counters)

**Example use cases:**
- Cache user profiles to avoid repeated DB reads
- Store JWT refresh tokens
- Rate limit counters per IP

---

#### `internal/platform/logger/logger.go`

**Purpose:** Initialises and exposes the application-wide structured logger using `zap`.

**What it does:**
- Creates a `zap.Logger` configured for the current environment (development = human-readable, production = JSON)
- Exposes `Info`, `Error`, `Warn`, `Debug`, `Fatal` helpers
- Attaches request ID, user ID, and other context fields automatically

**Why zap?** Much faster than `log` or `logrus` — zero-allocation logging matters at scale.

**Used by:** All middleware, all handlers, all services when they need to log events.

---

#### `internal/platform/middleware/jwt.go`

**Purpose:** Gin middleware that authenticates every protected request.

**What it does:**
1. Reads the `Authorization: Bearer <token>` header
2. Validates the JWT signature and expiry
3. Extracts the user ID and role from the claims
4. Injects both into the Gin context for downstream handlers

**Used by:** All protected route groups in every module's `routes.go`

---

#### `internal/platform/middleware/authorize.go`

**Purpose:** Gin middleware for role-based authorization (RBAC).

**What it does:**
- Reads the user's role from the Gin context (set by `jwt.go`)
- Checks if that role is in the list of allowed roles for the route
- Returns `403 Forbidden` if the role is insufficient

**Usage in routes:**
```go
admin := v1.Group("/admin")
admin.Use(middleware.JWTAuth(authSvc))
admin.Use(middleware.Authorize("admin"))
{
    admin.DELETE("/users/:id", userHandler.AdminDeleteUser)
}
```

---

#### `internal/platform/middleware/ratelimit.go`

**Purpose:** Limits how many requests a single IP can make in a time window.

**What it does:**
- Uses Redis to track request counts per IP
- Returns `429 Too Many Requests` when the limit is exceeded
- Window and limit are configurable (e.g. 100 requests per minute)

**Why Redis?** In-memory rate limiting breaks with multiple server instances. Redis is shared across all instances so the limit is enforced globally.

---

#### `internal/platform/middleware/logger.go`

**Purpose:** Logs every incoming request and its response.

**What it logs:**
- Method, path, status code, latency
- Request ID (generated per request)
- User ID (if authenticated)
- Client IP

**Why separate from the app logger?** The request logger runs on every single request. Keeping it separate means you can tune its log level independently.

---

#### `internal/platform/middleware/metrics.go`

**Purpose:** Records Prometheus metrics for every request.

**What it records:**
- Request count by method, path, status code
- Request duration histogram
- Active connections gauge

---

#### `internal/platform/metrics/prometheus.go`

**Purpose:** Defines and registers Prometheus metric collectors, and exposes the `/metrics` endpoint.

**What it does:**
- Registers `http_requests_total`, `http_request_duration_seconds`, etc.
- Mounts `GET /metrics` on the Gin engine using the `promhttp` handler
- Grafana scrapes this endpoint to build dashboards

---

### `pkg/`

Pure utility packages with no imports from `internal/`. Safe to import from anywhere in the project, including `platform/`.

| Package | Purpose |
|---|---|
| `config/` | Reads all environment variables into a typed `Config` struct. Builds the DSN. Uses `viper` for flexibility. |
| `response/` | Standard JSON envelope: `{"success": bool, "message": "...", "data": {...}}`. Every handler uses this. |
| `validator/` | Wraps `go-playground/validator` errors into structured per-field messages. |
| `errors/` | Defines app-level sentinel errors: `ErrNotFound`, `ErrForbidden`, `ErrConflict`, `ErrUnauthorized`. Maps them to HTTP status codes. |
| `pagination/` | Parses `?page=1&limit=20` query params. Calculates offset. Returns a `Pagination` struct for repository calls. |
| `crypto/` | Wraps `bcrypt` for password hashing. Wraps `crypto/rand` for generating secure tokens. |

---

### `db/`

Everything database-related that is not Go code.

#### `db/migrations/`

Plain SQL files managed by `golang-migrate`. Naming convention:

```
{version}_{description}.up.sql     ← applies the change
{version}_{description}.down.sql   ← reverts the change
```

**Rules:**
- Never edit a migration file after it has been applied to any environment
- Always write a matching `.down.sql` for every `.up.sql`
- Each migration does one thing (one table, one index, one column)

#### `db/queries/`

SQL query files read by `sqlc`. Each file corresponds to one database table.

Annotation syntax:
```sql
-- name: FunctionName :one   → returns one row
-- name: FunctionName :many  → returns a slice
-- name: FunctionName :exec  → returns only an error
```

After editing any file here, run `sqlc generate` from the project root.

#### `db/sqlc/`

Auto-generated by `sqlc`. Never edit manually. Contains:
- `db.go` — the `Queries` struct and `DBTX` interface
- `models.go` — Go structs mirroring database tables
- `*.sql.go` — one typed Go function per SQL query

---

### `test/integration/`

**Purpose:** Integration tests that run against a real database and Redis instance.

**How they differ from unit tests:**
- Unit tests (`*_test.go` next to source files) use mock repositories — no real database
- Integration tests spin up real infrastructure (via Docker in CI) and test the full stack from service down to database

**What they test:**
- That SQL queries return the right data
- That transactions roll back correctly on error
- That cache invalidation works as expected

---

### `deployments/`

#### `docker-compose.yml`

Local development stack. Runs postgres, Redis, the migrate container, and the app together.

```
postgres  →  migrate (runs up)  →  app
redis     →  (app connects on startup)
```

#### `docker-compose.prod.yml`

Production overrides. Points to real credentials via Docker secrets, sets `GIN_MODE=release`, removes debug ports.

#### `deployments/grafana/dashboards/app.json`

Pre-built Grafana dashboard JSON. Imported automatically when the Grafana container starts. Displays request rate, error rate, latency percentiles, and active connections from Prometheus data.

---

### `.github/workflows/`

#### `ci.yml` — runs on every push

```
checkout → setup Go → go mod download → sqlc verify → go build → migrate → go test
```

Spins up a real PostgreSQL and Redis service container so integration tests have real infrastructure.

#### `cd.yml` — runs on push to `main` only, after CI passes

```
build multi-arch Docker image (amd64 + arm64)
  → push to Docker Hub
    → SSH into server
      → docker compose run migrate
        → docker compose up --no-deps app
```

---

### `Dockerfile`

Multi-stage build:
1. `golang:1.21-alpine` — builds the binary with `CGO_ENABLED=0`
2. `alpine:latest` — copies only the binary, no Go toolchain

Result is a minimal image (~15MB) with no build tools.

---

### `Makefile`

Developer shortcuts so you never need to memorise long commands:

```makefile
make run          # go run ./cmd/api
make build        # go build -o bin/api ./cmd/api
make test         # go test ./... -v -race
make migrate-up   # apply all pending migrations
make migrate-down # roll back one migration
make sqlc         # sqlc generate
make docker-up    # docker compose up --build
make docker-down  # docker compose down
make lint         # golangci-lint run
```

---

### `sqlc.yaml`

Tells sqlc:
- Which SQL engine (postgresql)
- Where migrations live (schema source)
- Where query files live
- Where to write generated Go code
- Type overrides: `uuid → github.com/google/uuid`, `timestamptz → time.Time`

---

### `.env.example`

Documents every environment variable the app needs. Copy to `.env` and fill in real values before running locally.

```
PORT, GIN_MODE, JWT_SECRET, TOKEN_TTL
DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME, DB_SSLMODE
REDIS_HOST, REDIS_PORT, REDIS_PASSWORD
```

## How to add a new business module

Say you want to add an `order` module:

1. Create `internal/order/`
2. Write these 5 files in order:
   - `model.go` — define your structs
   - `repository.go` — implement `OrderRepository` interface using sqlc
   - `service.go` — implement `OrderService` interface with business logic
   - `handler.go` — implement HTTP handlers calling the service
   - `routes.go` — register routes on the Gin engine
3. Add SQL migration: `db/migrations/003_create_orders.up.sql`
4. Add SQL queries: `db/queries/orders.sql`
5. Run `sqlc generate`
6. Wire in `cmd/api/main.go` — instantiate repo → service → handler → routes

No other files need to change. The other modules are completely unaffected.
